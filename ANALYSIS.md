# Анализ проекта `Image Pusser`

## 1) Что делает проект

Приложение — это desktop-утилита на **Python + Eel** с web-интерфейсом.
Основной поток:

1. Берёт строки запросов из `Key.txt`.
2. Ищет ссылки на изображения через Google Images HTML.
3. Скачивает картинки в папки `1`, `2`, ...
4. Фильтрует по размеру, удаляет дубли (dHash + резкость).
5. Помечает подозрительные картинки OCR-ом (watermark/текст).
6. Даёт вручную отметить картинки на удаление в UI.
7. Делает постобработку (приведение к 1920x1080, боке-фон для узких кадров).

## 2) Сильные стороны

- **Хорошо разделены этапы пайплайна**: загрузка, дедупликация, OCR-флаги, постобработка.
- **Неплохая UX-логика в UI**: live-thumbs, навигация по “страницам” (папкам), прогресс-бары, предпросмотр.
- **Защита от мусора на входе**: минимальный размер файлов, минимальное разрешение, конвертация PNG/WEBP в JPEG.
- **Практичный анти-дубль**: perceptual hash + выбор более резкого кадра.
- **Динамический список блокируемых доменов** с сохранением в файл.

## 3) Ключевые риски и потенциальные баги

### 3.1 Безопасность/стабильность загрузки

- В `download_one()` нет ограничения `Content-Length` и общего лимита на размер тела ответа.
  - Риск: скачивание очень больших файлов (DoS по диску/ОЗУ/времени).
  - Что сделать: проверять `Content-Length` (если есть) и обрывать поток после, например, 15–25 МБ.

- Все `except:` слишком широкие и “глотают” исключения.
  - Риск: сложно дебажить, тихие отказы маскируются как нормальное поведение.
  - Что сделать: заменить на `except Exception as e:` + лог с контекстом.

### 3.2 OCR и производительность

- `easyocr.Reader(..., gpu=True)` жёстко просит GPU.
  - Риск: на CPU-only системах инициализация может падать или сильно тормозить.
  - Что сделать: пытаться `gpu=True`, при неудаче fallback на `gpu=False`.

- OCR-воркер один на процесс — это хорошо для контроля нагрузки, но на больших партиях может стать узким местом.
  - Возможный апгрейд: очередь с регулируемым пулом OCR-воркеров (1..N) с конфигом.

### 3.3 Логика дедупликации

- Сейчас дедупликация O(N²) по числу файлов в папке.
  - Для сотен файлов ок, для тысяч станет тяжело.
  - Оптимизация: LSH/бакетизация по префиксу hash либо ранняя фильтрация по метаданным (размер/соотношение сторон).

### 3.4 Интеграция с Google Images

- Парсинг HTML регуляркой (`re.findall`) по конкретному паттерну.
  - Риск: Google меняет разметку/формат выдачи → резко падает число ссылок.
  - Что сделать: предусмотреть fallback-стратегии, более устойчивые паттерны, ретраи с разными заголовками.

### 3.5 Потоки и состояние

- `pages_data` меняется в фоне и одновременно читается из Eel-методов.
  - В CPython GIL частично сглаживает гонки, но логически состояние всё равно может быть “плавающим”.
  - Что сделать: добавить `threading.Lock` вокруг операций чтения/записи shared-state.

- Очередь OCR глобальная и завершение воркера через `None` не используется.
  - Это не баг, но “незакрытый” поток в вечном цикле.

## 4) Что хорошо в фронтенде

- Чистая и понятная структура (`index.html` + `script.js` + `style.css`).
- CSS аккуратно учитывает узкие места flex-layout (`min-height: 0`, `overflow`), поэтому интерфейс стабильный.
- Логика рендера карточек и marked-state читаемая.

## 5) UX и продуктовые улучшения

- Добавить настройки в UI:
  - количество потоков,
  - минимальные размеры,
  - порог дубликатов (Hamming),
  - включение/выключение OCR.
- Кнопка **Pause/Resume** для долгих задач.
- Экспорт отчёта по итогу: сколько ссылок найдено/скачано/отбраковано по причинам.
- Опция “не удалять сразу”, а перемещать в корзину проекта.

## 6) Быстрые технические улучшения (высокий ROI)

1. Ввести структурированное логирование (уровни: info/warn/error).
2. Ограничить максимальный размер скачиваемого файла.
3. Сузить `except:` до `except Exception as e` и писать причины в лог.
4. Добавить lock для `pages_data` и аккуратный доступ из Eel API.
5. Сделать OCR fallback GPU→CPU.

## 7) Общая оценка

Код решает практическую задачу и уже содержит полезные оптимизации для продакшен-потока (фильтры, дедуп, OCR-пометки, UI-контроль). Главные зоны риска — надёжность сети/парсинга, “тихие” исключения и потокобезопасность общего состояния. С небольшим рефакторингом проект можно сделать заметно устойчивее без потери скорости разработки.
